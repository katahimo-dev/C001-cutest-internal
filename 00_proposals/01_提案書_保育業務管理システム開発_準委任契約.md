## 2. ご提供サービス内容

### アーキテクチャ方針：モジュラー設計

本プロジェクトでは、**モジュラー設計**を採用します。
各モジュールは独立したコンポーネントとして開発し、将来的に顧客ごとに必要な機能だけを組み合わせて提供できる柔軟なシステムを構築します。

```

  【認証基盤（Auth Module）】 ※ 全モジュール共通
   ├── スタッフログイン・認証
   ├── 権限管理（管理者/スタッフ）
   ├── セッション管理（7日間有効期限）
   └── パスワードリセット・変更

  【データ基盤（Data Module）】 ※ 全モジュール共通
   ├── スタッフマスタ
   ├── 顧客マスタ（手動登録 or 外部連携）
   ├── スケジュール管理（手動登録 or 外部連携）
   └── 操作ログ・監査証跡

  【インポートアダプター】 ※ データ入力元（顧客の運用に応じて選択）

  【機能モジュール】 ※ 顧客ごとに選択可能

```
【現行運用】                          【新システム】
  RESERVA → CSV export               RESERVA → CSV → 顧客マスタ（自動インポート）
  Googleカレンダー（手動確認）         Googleカレンダー → スケジュール（自動同期）
                                               ↓
  LINE WORKS で経路情報を確認          PWA Push通知 → アプリ内で訪問先確認

**新システムのメリット：**
- LINE WORKSの契約・設定・運用が不要になる

- 出勤簿をスタッフ自身が修正可能
- 管理者は社労士向けCSVを一括ダウンロード

### 各モジュールの詳細

#### 認証基盤モジュール（Auth Module）

現行GASの認証部分を強化し、全モジュールで共通利用できる認証基盤を構築します。

| 機能 | 現行GAS | 新システム |
|------|---------|-----------| 
| ログイン方式 | メールアドレス + パスワード | 同左（Firebase Authentication） |
| パスワード保存 | Salt付きSHA-256ハッシュ | Firebase標準（bcrypt相当） |
| セッション管理 | 7日間ローリング有効期限 | 同左（Firebase Auth Token） |
| パスワードリセット | メール認証コード | 同左（Firebase標準機能） |
| 二段階認証 | 未対応 | **対応可能**（オプション） |

#### データ基盤モジュール（Data Module）

スタッフ・顧客・スケジュール情報を一元管理する共通データ基盤です。

| 機能 | 概要 |
|------|------|
| **スタッフマスタ** | 氏名、住所、連絡先、権限等（手動登録 or CSVインポート） |
| **顧客マスタ** | 顧客情報の一元管理（手動登録 or 外部連携アダプター経由） |
| **スケジュール管理** | 訪問予定の管理（手動登録 or 外部連携アダプター経由） |
| **操作ログ** | 誰が・いつ・何をしたか記録（監査証跡） |

```
【データ取り込みパターン】

パターンA（キューテスト社）:
  RESERVA CSV → 顧客マスタ
  Googleカレンダー → スケジュール管理

パターンB（他社例）:
  手動登録 → 顧客マスタ
  手動登録 → スケジュール管理

パターンC（将来）:
  予約機能モジュール → 顧客マスタ + スケジュール管理
```

#### 移動経路モジュール（Route Module）

現行の「ルート検索」GASを移行・強化します。

| 機能 | 概要 |
|------|------|
| **経路自動検索** | Google Maps API連携、毎日18:00-19:00に翌日分を検索 |
| **PWA Push通知** | 検索結果をスタッフへPush通知（外部サービス不要） |

現行の「出勤簿管理シート」GASを移行・強化します。
**Webアプリ上で各スタッフが直接編集**でき、管理者は一括エクスポート機能で社労士向けデータを出力します。

| 機能 | 概要 |
|------|------|
| **出勤簿自動生成** | スケジュールから自動生成（Firestoreに保存） |
| **スタッフ編集画面** | 各スタッフがアプリ上で始業/終業時刻などを直接修正 |

| **管理者ダッシュボード** | 全スタッフの出勤状況を一覧表示・承認機能 |
| **一括エクスポート** | 管理者がスタッフごとのCSV/Excelを一括ダウンロード |
| **自動バックアップ** | 月末所定日に自動でGoogle Driveへ保存（社労士共有用） |

#### 日報モジュール（Report Module）

現行の「日報自動成型アプリ」GASを移行・強化します。
**日報は全スタッフで情報共有**する運用に対応し、PWA Push通知で即時配信します。

| 機能 | 概要 |
|------|------|
| **AI日報生成** | Gemini APIで「社内向け」「保護者向け」2種類を自動生成 |
| **事故報告** | ヒヤリハット含む事故報告書の自動整形 |
| **領収書管理** | カメラ撮影/アルバム選択で画像登録、一覧管理 |
| **PWA Push通知** | 日報・事故報告を全スタッフへ即時通知（情報共有用） |
| **通知サイレント設定** | 通知種別ごとにサイレント（音なし）設定可能 |
| **顧客別履歴** | 過去の日報・事故報告を顧客別に検索・閲覧 |
| **日付別履歴** | 日付ごとに全顧客横断で活動記録を一覧表示 |
| **顧客情報閲覧** | データ基盤の顧客マスタを参照 |
| **匿名化モード** | テスト・デモ用に個人情報を匿名化（PoCモード） |

```
【現行システム】
  スタッフが日報送信 → LINE WORKSトークルームに投稿

【日付別履歴のメリット】

  日報未提出のスタッフを即座に発見
  事故報告の見落とし防止

### 技術スタック

本プロジェクトでは、**Googleが提供する最新のクラウド技術**を中心に採用します。
これにより、高い信頼性・拡張性・コストパフォーマンスを実現します。

| 技術 | 採用理由・メリット |
|------|-------------------|
| **React 18** | Facebook（Meta）が開発した世界シェアNo.1のUI構築技術。高速な画面表示と快適な操作感を実現。技術者が多く、将来の保守・引き継ぎも容易 |
| **TypeScript** | Microsoft製の型安全な言語。プログラムのバグを事前に検出し、品質の高いコードを効率的に開発可能 |
| **Material-UI** | Googleのデザインガイドラインに準拠した美しいUI部品集。スマホ・PC両対応のデザインを効率的に実現 |
| **Firebase** | **Googleが提供するクラウドサービス基盤**。サーバー管理不要で、自動的にアクセス増加に対応。世界中の大企業が採用（Spotify、The New York Times等） |
| **Cloud Firestore** | Googleのデータベース。リアルタイム同期対応で、複数端末から同時アクセスしても即座に反映。自動バックアップ付き |
| **Firebase Authentication** | Googleの認証システム。パスワード管理・不正アクセス防止をGoogleが担保。二段階認証にも対応 |
| **Firebase Functions** | サーバーレスでバックエンド処理を実行。モジュール単位でデプロイ可能で、マイクロサービス的な構成を実現 |
| **Gemini API** | **GoogleのAI（人工知能）**。日報の自動生成・事故報告書作成を高精度で実現 |
| **PWA対応** | Webサイトをスマホアプリのように使える技術。**Push通知対応**で外部サービス不要。App Store申請不要で、ホーム画面に追加してアプリ感覚で利用可能 |

```
技術選定のポイント:
  全てGoogle製品で統一 → 連携がスムーズ、サポートが充実
  世界標準の技術 → 将来の機能追加・保守が容易
  サーバー管理不要 → 運用コストを最小化
  従量課金制 → 使った分だけ支払い、無駄がない
```

#### モジュラー設計を支える技術構成

```
【フロントエンド】
  └─ React 18 + TypeScript
      └─ 機能ごとにコンポーネント分離
          ├─ @/features/auth/        ... 認証機能
          ├─ @/features/data/        ... データ管理（スタッフ/顧客/スケジュール）
          ├─ @/features/route/       ... 移動経路機能
          ├─ @/features/attendance/  ... 出勤簿機能
          └─ @/features/report/      ... 日報機能

【バックエンド（Firebase Functions）】
  └─ 機能ごとに独立したFunction群
      ├─ auth-*       ... 認証関連API
      ├─ data-*       ... データ管理API
      ├─ import-*     ... インポートアダプター（差し替え可能）
      │   ├─ import-reserva-*      ... RESERVA CSV取込
      │   └─ import-gcal-*         ... Googleカレンダー連携
      ├─ push-*       ... PWA Push通知（標準機能）
      │   ├─ push-route-*          ... 翌日訪問先通知
      │   ├─ push-report-*         ... 日報リマインダー
      │   └─ push-alert-*          ... 事故報告アラート
      ├─ export-*     ... エクスポート機能
      │   └─ export-attendance-*   ... 出勤簿CSV一括ダウンロード/Google Drive自動保存
      ├─ route-*      ... 移動経路検索
      ├─ attendance-* ... 出勤簿自動生成
      └─ report-*     ... 日報生成・保存

【データベース（Firestore）】
  └─ コレクション単位でモジュール分離
      ├─ /users       ... スタッフ情報（共通）
      ├─ /customers   ... 顧客情報（共通）
      ├─ /schedules   ... スケジュール情報（共通）
      ├─ /routes      ... 経路検索結果
      ├─ /attendance  ... 出勤簿データ
      └─ /reports     ... 日報・事故報告
```

この構成により：
- **モジュール単位での機能ON/OFF** が容易
- **顧客ごとに異なる機能セット** を提供可能
- **外部連携アダプターの差し替え** が容易（RESERVA→別システムへの移行等）
- **独立したデプロイ・テスト** でリスク分散
- **将来の機能追加** が既存機能に影響しない

## 11. パートナーシップ戦略：SaaS事業化と収益モデル

本プロジェクトは、単なる受託開発にとどまらず、将来的な**SaaS（Software as a Service）事業**への発展を見据えた戦略的パートナーシップをご提案します。

個人事業主である開発者（かたひも）と、法人である貴社（株式会社キューテスト）が、それぞれの強みを活かして新たな収益源を獲得する**「貴社主体」の事業モデル**を構築します。

### 事業構造：貴社をSaaS提供主体とするモデル

**株式会社キューテスト**をサービス提供主体（パブリッシャー）、**かたひも**を技術パートナー（ベンダー）と位置づけます。

```
┌─────────────────────────────────────────────────────────┐
│  【サービス提供主体】 株式会社キューテスト                  │
│  ・SaaSブランドオーナー、販売元                            │
│  ・エンドユーザー（他ベビーシッター法人）との契約主体                  │
│  ・一次問い合わせ対応、営業活動                            │
└─────────────────────────────────────────────────────────┘
            ↑  ↓
            │  ① サービス提供
            │  ② 利用料受領
            │
┌──────────────────────────────┐
│  【エンドユーザー】 ベビーシッター法人  │
└──────────────────────────────┘
```

**開発者（かたひも）の役割：**
- 株式会社キューテストに対する技術提供（システム開発・保守）
- SaaSの継続的な機能エンハンス
- インフラ運用サポート

この構造により、**貴社はサービスの販売・ブランド構築に、開発者は技術品質の担保に注力する「最適な役割分担」**を実現します。

### 成長戦略：「ミニマムSaaS」による早期収益化

最初からフル機能を完成させて販売するのではなく、**「ミニマムSaaS（必要最小限の機能）」を早期にリリース**し、その収益を開発費に再投資するサイクルを作ります。

```
【ステップ1：ミニマムSaaSリリース】（開発6〜8ヶ月目）
  ・貴社で実証済みの「日報・通知機能」等のコア機能のみを他社へ提供
  ・安価なプランで導入ハードルを下げる
  ・獲得した収益を「出勤簿」や「高度な機能」の開発費に充当
        ↓
【ステップ2：機能拡張と単価アップ】
  ・収益を原資に新機能を開発
  ・既存ユーザーへのアップセル、上位プランの提供
  ・機能充実によりSaaSとしての市場価値向上
```

この戦略により、**貴社の持ち出し費用（初期投資）を最小限に抑えつつ、事業を拡大**することが可能です。

### 収益分配モデル案

貴社がSaaSとして他社に販売し得た収益の一部を、**技術利用料**として開発者に還元いただくモデルを提案します。

| 項目 | 貴社（キューテスト） | 開発者（かたひも） | 備考 |
|------|--------------------|-------------------|------|
| **役割** | 販売・契約・一次サポ | 開発・保守・技術支援 | |
| **収益分配** | **売上の 50〜70%** | **売上の 30〜50%** | サーバー費等の実費を除いた粗利で分配 |
| **名目** | 事業収益 | 技術ライセンス料 | |

**貴社のメリット：**
1.  **初期開発費の回収**：SaaS収益により、本プロジェクトの委任契約費用を早期に回収可能
2.  **新たな収益の柱**：自社利用のために作ったシステムが、継続的な利益を生む資産に変わる
3.  **維持費の相殺**：得られた収益で、自社利用分のシステム維持費を実質ゼロにできる

**具体的な分配率や条件は、SaaS事業化のフェーズ（ステップ1）開始時に改めて協議・決定します。**

### なぜこのモデルなのか

- **責任分界の明確化**: 貴社が事業主体となることで、対外的な信用力が向上し、販売しやすくなります。
- **持続可能性**: 開発者にも収益還元されることで、モチベーション維持と長期的な技術サポートが保証されます。
- **共創関係**: 「発注者と受注者」を超えた、同じゴール（システムの実用化と普及）を目指すパートナーとなります。

## 12. 事業継続性確保の仕組み

### 個人事業主への発注リスクと対策

開発者が個人事業主であることに伴うリスクを正面から受け止め、**万一の場合でも貴社の事業が止まらない仕組み**を構築します。

| リスク | 発生確率 | 影響度 | 対策 |
|--------|----------|--------|------|

| 開発者との連絡途絶 | 低 | 高 | GitHub/Firebase共有、30日ルール |
| 開発者の死亡 | 極低 | 最高 | エスクロー体制、緊急連絡先の代理権限 |
| 情報漏洩 | 低 | 高 | 賠償責任保険、セキュリティ対策 |
| 開発品質の問題 | 中 | 中 | 6ヶ月瑕疵保証、週次レビュー |

### 対策1：コード・インフラの共有体制

```
**共有体制の全体像**

- 【GitHub リポジトリ】
    - オーナー: 開発者
    - Collaborator: キューテスト社担当者
        - 常時コードにアクセス可能
        - 開発者不在でも他者への引き継ぎ可能

- 【Firebase プロジェクト】
    - オーナー: 開発者（メイン管理）
    - オーナー: キューテスト社（バックアップ）
    - 課金: キューテスト社法人カード
        - 両者とも完全な管理権限
        - 開発者不在でもシステム継続運用可能

- 【ドキュメント】
    - Google Drive で共有
    - キューテスト社に編集権限
        - 運用マニュアル・設計書に常時アクセス可能
```

### 対策2：緊急連絡先の登録

開発者は、緊急事態発生時に貴社への連絡が可能な**緊急連絡先（親族等）**を登録します。

| 項目 | 内容 |
|------|------|
| 登録対象 | 親族・信頼できる知人 |
| 登録情報 | 氏名、電話番号、メールアドレス |

### 対策3：30日ルール

開発者が**30日以上連絡不能**となった場合の対応手順を明文化します。

```
【30日ルール：対応フロー】

Day 1-7   : 通常の連絡手段（メール・電話）で連絡試行
Day 8-14  : 緊急連絡先への電話・メールによる連絡
Day 15-29 : 状況確認・対応協議
Day 30    : 「緊急事態」と判定
            ↓
        以下の措置を発動:
        ・GitHub リポジトリの管理権限取得
        ・Firebase プロジェクトの単独管理開始
        ・他の開発者への引き継ぎ検討開始
