# gas-root-serach 詳細仕様書

## 1. 概要
本プロジェクトは、Googleカレンダーの予定からスタッフの訪問スケジュールを解析し、最適な移動ルート（時間・距離）を算出、記録するとともに、LineWorksを通じてスタッフへ毎日の予定を通知する統合システムです。

## 2. 主な機能
### 2.1. 予定データの抽出・解析
- **カレンダー連携**: 会社全体のGoogleカレンダーを走査し、各スタッフの予定を取得します。
- **予定種別の自動判定**: タイトルに含まれるタグに基づき、以下の種別を自動判別します：
    - `[予約確定]`: 通常の顧客訪問（本文の「施設：〇〇」からスタッフを特定）。
    - `[新規]`: 新規顧客。場所情報から座標を取得。
    - `[イベント]`: 会議等。招待ゲストリストからスタッフを特定。
    - `[事務]`: 移動計算の対象外とする事務作業。
- **オンライン対応**: 説明文に「オンライン」が含まれる場合は移動計算から除外します。

### 2.2. ルート計算エンジン
- **Google Maps API 連携**: 自宅・訪問先・次の訪問先間の移動を計算します。
    - **出勤経路**: 自宅 → 1件目の訪問先
    - **移動経路**: N件目の訪問先 → N+1件目の訪問先
    - **退勤経路**: 最後の訪問先 → 自宅
- **期間限定住所の処理**: スタッフや顧客に「期間限定の住所（住所2）」が設定されている場合、指定された期間内は優先的にそちらをルート計算に使用します。
- **ジオコーディング**: 住所文字列から緯度経度が取得できない場合、自動的にGoogle Mapsのジオコーダーで補完します。

### 2.3. データ集計・出力
- **ルート集計への書き込み**: 計算された時間（分）、距離（km）、及びGoogleマップのリンクを、月別のシート（例: `202601`）に1行ずつ追記出力します。

### 2.4. 通知機能 (`lineworks.js`)
- **LineWorks Bot 連携**: 解析されたスケジュールに基づき、スタッフごとにパーソナライズされた訪問リスト（時間、顧客名、URL等）をLineWorksで自動送信します。
- **認証管理**: JWTを使用したService Account認証によりLineWorks APIと通信します。

## 3. システム構成
- **入力ソース**: Google Calendar, 顧客CSV (Drive内 Kokyaku_...), スタッフマスタースプレッドシート。
- **計算エンジン**: Google Maps Directions Service / Geocoding Service.
- **出力先**: 「ルート集計」スプレッドシート。
- **外部通知**: LineWorks Messaging API.

## 4. 主要な設定 (`CONFIG`)
- `CUSTOMER_FOLDER_ID`: 顧客CSVが保存されているフォルダのID。
- `STAFF_SS_ID`: スタッフ情報のマスターシートID。
- `LW`: LineWorks API連携用の認証情報（Script Propertiesから取得）。

## 5. 運用方法
- **自動実行**: 毎日夜間にトリガーを実行し、翌日の全スタッフのルート計算と通知を自動で行います。
- **手動実行**: `runSpecifiedDate()` 関数により、特定の日付を指定して再計算が可能です。

## 6. 勤怠集計運用（2026-04 追加）
### 6.1. 勤怠集計ファイル
- ルート集計とは別に、勤怠転記専用のファイル「勤怠集計」を作成・更新する。
- 保存先フォルダは ID: `1FA2aSBddgBakETEbzJhJIx1vWG06P46J`。
- 月別シート（例: 202603）に、種別（CUSTOMER APPOINTMENT / OFFICE WORK / EVENT）付きで保存する。

### 6.2. トリガー設定
- 関数名: `autoRunSaveAttendance()`
- 実行時刻: 毎日 17:00〜18:00（当日分データを夕方更新）
- この処理では LineWorks 通知は実施しない。

### 6.3. デバッグ実行（2026年3月４月）
- 一括: `debugSaveAttendanceMarch2026()`
- 一括: `debugSaveAttendanceApril2026()`

### 6.4. 運用上の注意
- Maps API のルート計算件数が多い場合、月一括処理はタイムアウトする可能性がある。
- タイムアウト回避が必要な場合は、週単位関数を個別実行する。
- 1日の最初が事務作業または15分顧客対応のみの場合、訪問の出勤距離（AI列相当）が空欄になる挙動は仕様として許容する。
